Execute tasks from specs, plans, or direct instructions using wave-based parallel execution with fresh context per task.

## Execution Model

### Wave-Based Parallelism

Tasks are grouped into waves based on dependencies. Independent tasks run as parallel subagents. Each subagent gets fresh context (no accumulated garbage from prior tasks).

```
WAVE 1 (parallel)         WAVE 2 (parallel)        WAVE 3
┌──────────┐ ┌──────────┐  ┌──────────┐ ┌──────────┐  ┌──────────┐
│ DB Schema│ │ Auth Lib │  │ API Routes│ │ Middleware│  │ UI Pages │
└──────────┘ └──────────┘  └──────────┘ └──────────┘  └──────────┘
     no deps                  needs wave 1              needs wave 2
```

### Fresh Context Per Task

Each task runs as an Agent subagent. This means:
- Full 200k context window dedicated to that task alone
- No quality degradation from prior task context
- Parallel tasks don't interfere with each other
- Main conversation stays clean for orchestration

---

## Phase 1: Task Extraction

Read the source material and extract atomic tasks:

1. Check for existing specs/plans:
   - `specs/*.md` or `plans/*.md` from `/spec` or `/plan`
   - `.claude-suite/quality/*/tasks.md` from `/analyze` (if `--quality`)
   - Direct user instructions in the conversation

2. For each task, identify:
   - **What**: Clear deliverable (file created, feature working, test passing)
   - **Files**: Which files will be created or modified
   - **Depends on**: Which other tasks must complete first (by file overlap or logical dependency)
   - **Verify**: How to confirm the task is done (build passes, test passes, endpoint responds)

3. Present the task list to the user for approval before executing.

---

## Phase 2: Wave Planning

Group tasks into waves based on dependencies:

**Rules:**
- Tasks with no dependencies → Wave 1
- Tasks depending on Wave 1 outputs → Wave 2
- Tasks depending on Wave 2 outputs → Wave 3
- Tasks that modify the SAME files → same wave, sequential (prevent conflicts)

**Show the user the wave plan:**

```
Wave 1 (parallel): Task 1, Task 2, Task 3
Wave 2 (parallel): Task 4, Task 5
Wave 3 (sequential): Task 6
```

If `--dry-run`, stop here.

---

## Phase 3: Wave Execution

For each wave:

### 3.1 Launch Parallel Subagents

For each task in the wave, spawn an Agent with:
- The task description, file list, and verification steps
- Relevant existing code context (read key files first, include in prompt)
- Project conventions from CLAUDE.md / memory
- Instruction to commit atomically when done

**Agent prompt template:**
```
You are executing a single development task. Focus only on this task.

PROJECT CONTEXT:
[Include relevant CLAUDE.md snippets, design system, auth patterns]

TASK:
[Task description]

FILES TO CREATE/MODIFY:
[File list]

VERIFICATION:
[How to verify this task is complete]

INSTRUCTIONS:
1. Read any existing files you need to understand
2. Implement the task
3. Run the verification step (build, test, etc.)
4. If verification fails, fix and retry
5. Commit with message: "feat: [task description]"
```

Use `mode: "auto"` for subagents so they can execute without permission prompts.

### 3.2 Wait for Wave Completion

All tasks in a wave must complete before the next wave starts.
If a task fails, report the error and ask the user whether to:
- Retry the failed task
- Skip it and continue
- Stop execution

### 3.3 Verify Wave

After each wave completes:
- Run `npm run build` (or equivalent) to catch cross-task integration issues
- If build breaks, identify which task caused it and fix before next wave

---

## Phase 4: Final Verification

After all waves complete:

1. Run full build: `npm run build`
2. Run tests if they exist: `npm test`
3. Summarize what was built:
   - Files created/modified
   - Tasks completed
   - Any issues encountered
4. List any manual verification the user should do

## --quality Flag: Execute Quality Tasks

When `/execute --quality` is used:

### Task Source
Reads tasks from `.claude-suite/quality/*/tasks.md` generated by `/analyze`

### Execution Flow

1. **Load Current Tasks**
   - Find most recent analysis
   - Show task progress
   - Identify next task

2. **Execute via Waves**
   - Group quality tasks by dependency (security fixes first, then bugs, then improvements)
   - Each task gets its own subagent with fresh context
   - Atomic commits per task

3. **Update Progress**
   - Check off completed items
   - Update progress.md
   - Calculate new metrics

---

## --sequential Flag

Forces all tasks to run one at a time in the main context. Use when:
- Tasks are tightly coupled and hard to separate
- You want to watch each task execute
- The project is small enough that context rot isn't a concern

---

## Orchestrator Responsibilities

The main conversation (you, the orchestrator) should ONLY:
- Extract and order tasks
- Spawn subagents
- Monitor completion
- Handle failures
- Run cross-task verification (builds, tests)

NEVER do the implementation work in the main context. Always delegate to subagents. This keeps your context clean for the full session.
